Desktop concept — the Command Center same app, same engine, no schema change

The phone stays exactly as it is: a field tool, one thing at a time. Desktop stops being a stretched phone and becomes the office tool: everything on one screen, keyboard-first, plus the two things the phone can't do at all today — editing the price tables and seeing a price change's blast radius before saving. Built as a responsive layout (and a Prices route) in the same Next.js app — same API routes, same lib/pricing.ts, same cache.

1 · One screen, three panes

Projects list + project detail + price board side by side. No back-and-forth navigation — click a project on the left, it loads in the middle, the board updates on the right.

2 · Keyboard-first

⌘K jumps to any client. Measurements edit like a spreadsheet: click a cell, type, Tab, Enter for a new row. No forms, no bottom sheets.

3 · Price tables, finally editable

Today price changes go through SQL. A Prices screen edits fence / gate / extras / settings inline — with a live preview of which projects move and by how much.

Three panes, zero navigation. The phone's home → project → board flow becomes one screen. Clicking a project in the left rail swaps the center + right panes — and since the client cache already prefetches every bundle on entry, the swap is instant with no new server work.
Spreadsheet measurements. The grid is the desktop replacement for the section form: click a cell, type, Tab to the next, Enter to append a row (take-down still auto-mirrors linear ft). Each row keeps the computed price-under-active-fence column. Same PATCH endpoints as the phone forms.
What-if is read-only by design. Margin stays set-and-forget (your product rule) — the slider only recomputes the displayed total locally and resets when you leave. It never writes. If even that feels like clutter, it's the first thing to cut.
Open question — totals in the left rail. You deliberately keep totals off the phone home list. The office is a different context (no customer looking over your shoulder), so this concept shows them. Easy to drop if you want one rule everywhere.

The biggest single win. Price-table changes today go through Claude + SQL. Because every quote is computed on read, a price edit silently re-prices every open project — which is exactly why the phone shouldn't have this screen, and why desktop shows the blast radius first: the preview just runs the same engine against affected projects before you commit. Unpriced types stay loud. $0 rows carry the same ⚠ treatment as the board — and this screen is finally the place to fix them.

How it's built (and what it isn't)

Not a second app. One Next.js codebase, one deploy, one PIN. The three-pane layout is a min-width: 1024px media-query shell around the existing pages; below it, today's phone UI renders untouched. The pieces land independently, roughly in value order:

  1. Prices screen + blast radius — new route + small CRUD API routes; engine reused as-is.
  2. Three-pane shell — layout/CSS only; panes are the existing components rearranged, fed by the existing cache.
  3. Spreadsheet measurement grid — desktop-only component over the existing section PATCH routes.
  4. ⌘K palette + keyboard nav — client-only; the project list is already cached, so search is local.
  5. What-if panel — pure client math via the engine; optional, last.

Out of scope on purpose: no schema changes, no second auth story, no Electron wrapper, no phone-UI changes, and nothing the engine doesn't already compute.
