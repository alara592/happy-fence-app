All of it comes from data already in the browser. Today's visits = the appointments the calendar sync already stores, filtered to today. Needs-attention + card totals = the project bundles the home screen already prefetches into the cache (the bundle has the total; the phone list just chooses not to render it). Zero new server work, no schema changes.
Totals are desktop-only. The phone list keeps your no-dollars rule. Cards with no active fence show a quiet "no fence" instead of a number — same warning-not-$0 principle as everywhere else.
Needs attention = two rules, both cheap: a project with measurements but no active fence ("priced nothing yet"), and a project with no measurements at all (often a promoted Quick Quote or created-from-appointment shell). Click → straight to the project. When the list is empty the strip disappears entirely.
One judgment call I made: "Today" shows only today's visits (the dashboard is a morning view, not a planner — the week lives in Appointments). If you'd rather see today + tomorrow here, it's a one-line change. Flag it when you review.

How B differs from A — same ingredients, different hero. A reads top-to-bottom like a morning briefing: today first, then what's stuck, then the list. B keeps the LIST as the main surface (closer to today's app) and parks today + needs-attention + a pipeline number in a sidebar that stays put while you scroll projects. B is better if you're in the list constantly and glance at today; A is better if the first question every morning is "where am I going."
B adds one thing A doesn't have: the "On the board" pipeline number — total of all open quotes with an active fence. It's a one-liner from the same cached bundles, and it can be added to A (or dropped from B) freely. Cards drop to 2-across in B since the sidebar takes ~320px.
